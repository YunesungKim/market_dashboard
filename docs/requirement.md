## 0. 배경 및 범위

이 프로젝트는 **두 개의 독립된 Unit**으로 나뉜다. 두 Unit은 실행 환경과 배포 방식이 다르므로 각각 별도의 PRD 섹션으로 정의한다.

| 구분 | 실행 위치 | 산출물 |
|---|---|---|
| **Unit 1: 브리핑 생성 도구** | 로컬 PC (Flask 로컬 서버 + 브라우저 생성 화면) | 브리핑 데이터(JSON) 생성 + GitHub push |
| **Unit 2: 브리핑 아카이브** | GitHub Pages (공개 웹) | 카드 인덱스 + 상세 뷰 정적 사이트 |

> **참고 — GitHub Pages란?**
> GitHub 저장소 안의 정적 파일(HTML/CSS/JS)을 그대로 웹사이트로 서빙해주는 무료 호스팅 기능. 저장소 Settings → Pages에서 브랜치를 지정하면 `https://아이디.github.io/저장소명/` 주소로 공개된다. 빌드 과정이 필요 없어, 시간 제약이 있는 이번 워크샵에는 순수 HTML/JS/JSON만 사용하는 것을 권장한다.

## 1. 목적

매일 아침 국내/해외 증시의 주요 동향을 정리해주는 대시보드를 만든다. 특정 기업/키워드 기반의 브리핑을 생성하고, 이를 누적 기록으로 남겨 과거 브리핑도 다시 확인할 수 있는 공개 아카이브를 제공한다.

뉴스 검색은 **Serper(serper.dev)** 의 API를 이용한다. (무료 가입 시 2,500회 검색 제공)

---

## 2. 아키텍처 개요

시스템은 **3개 파트**로 구성된다. 비밀 키는 로컬 서버에만 존재하며, 공개 웹으로 절대 나가지 않는다.

```
[브라우저: 생성 화면]  ⇄  [로컬 서버 app.py (Flask)]  ⇄  Serper API / LLM API
   (localhost, 비공개)         │  .env에서 키 로드
                               │  briefings.json append
                               │  git push
                               ▼
                    [GitHub 저장소] → [GitHub Pages: 공개 아카이브]
                                        index.html + briefings.json (서버·키 없음)
```

- **로컬 서버 (`app.py`, Flask)** — `.env`의 키 보관, Serper 검색, LLM 요약, `briefings.json` append, git push 수행. 생성 화면(HTML/JS)을 서빙.
- **생성 화면 (브라우저 UI)** — 로컬 서버가 서빙. 입력·모드 선택·미리보기·배포. **키는 절대 브라우저로 내려가지 않는다.**
- **공개 아카이브 (GitHub Pages)** — `index.html` + `briefings.json` 정적 파일만으로 동작. 서버·키 없음.

## 3. Unit 1 — 브리핑 생성 도구 (로컬 PC)

### 3.1 구성

- **로컬 서버:** Python **Flask** 단일 스크립트(`app.py`). `python app.py`로 실행.
- **생성 화면:** 서버가 `GET /`로 서빙하는 단일 페이지(입력 → 미리보기 → 배포 상태 전환).

### 3.2 기능 요구사항

| ID | 요구사항 | 설명 |
|---|---|---|
| F1-1 | 기업/키워드 입력 | 생성 화면에서 특정 기업 이름과 키워드를 입력 |
| F1-2 | 뉴스 검색 | Serper News API로 입력값 기준 주요 기사 검색 (`POST /api/search`) |
| F1-3 | 브리핑 생성 | 모드 A(Serper 조립) / 모드 B(LLM 요약) 중 선택하여 브리핑 생성 |
| F1-4 | LLM provider 추상화 | 모드 B의 요약 제공자를 Claude 기본, 향후 Gemini 등으로 교체 가능하게 설계 |
| F1-5 | 미리보기 | 배포 전 생성 화면에서 브리핑 결과를 미리 확인 |
| F1-6 | 데이터 저장 | 확정된 브리핑을 `briefings.json`에 append (`POST /api/publish`) |
| F1-7 | 배포(Publish) | `briefings.json`을 GitHub 저장소로 push하여 Unit 2에 반영 |
| F1-8 | 자동 동향 후보 | 화면 진입 시 미국 증시 Top3 + 한국 증시 Top3(총 6장)를 자동 검색해 모드 A로 후보 카드 생성 (`GET /api/trends`) |
| F1-9 | 스테이징 보드 | 자동 후보 + 수동 검색 결과를 후보 카드로 누적 관리 (배포 전 임시 상태) |
| F1-10 | 카드 큐레이션 | 각 후보 카드를 삭제하거나 제목/요약/상세를 편집 |
| F1-11 | 배치 배포 | 보드에 남긴 카드들을 한 번에 `briefings.json`에 append + push |

### 3.3 브리핑 생성 모드

| 모드 | 설명 | 필요한 키 | 우선순위 |
|---|---|---|---|
| **A. Serper 조립 (기본)** | LLM 없이 Serper 뉴스 결과(제목·스니펫·URL·발표일)를 그대로 조립하여 브리핑으로 저장 | `SERPER_API_KEY` | 필수 |
| **B. LLM 요약 (확장)** | Serper 검색 결과를 LLM이 종합하여 제목·요약·상세를 생성 | `SERPER_API_KEY` + 해당 provider 키 | 확장 |

- 생성 화면 UI에서 A/B를 토글로 선택하며, B일 때는 provider를 선택한다.
- **provider 추상화:** 공통 `Summarizer` 인터페이스를 두고 `ClaudeSummarizer`(기본)를 먼저 구현, 향후 `GeminiSummarizer` 등을 추가한다. 사용 provider는 환경변수 `LLM_PROVIDER`(기본값 `claude`)로 지정한다.

### 3.4 Credential 관리 (.env)

- 모든 비밀 키는 프로젝트 루트의 **`.env`** 파일에서 관리하며, 로컬 서버만 읽는다.
- **`.gitignore`에 `.env`를 반드시 등록**하여 커밋/공개되지 않게 한다.
- `.env` 예시:
  ```
  SERPER_API_KEY=your-serper-key
  LLM_PROVIDER=claude
  ANTHROPIC_API_KEY=your-anthropic-key
  # 향후: GEMINI_API_KEY=your-gemini-key
  ```
- 키 없이도 실행 흐름을 이해할 수 있도록 `.env.example`(값 없는 템플릿)을 함께 제공한다.

### 3.5 데이터 스키마 (`briefings.json`)

```json
{
  "id": "2026-07-14-samsung-electronics",
  "date": "2026-07-14",
  "company": "삼성전자",
  "keyword": "HBM",
  "title": "삼성전자 HBM4 관련 주요 동향",
  "summary": "핵심 요약 1~2문장",
  "detail": "상세 브리핑 본문 (모드 B는 LLM 종합, 모드 A는 스니펫 조합)",
  "generator": { "mode": "serper|llm", "provider": "claude|null", "model": "claude-...|null" },
  "sources": [
    { "title": "기사 제목", "url": "https://...", "publishedDate": "2026-07-13", "source": "매체명" }
  ],
  "createdAt": "2026-07-14T09:00:00+09:00"
}
```

- 이 스키마 하나로 Unit 1(생성)과 Unit 2(렌더링)가 통신한다.
- 파일 하나(`briefings.json`)에 배열로 누적하는 방식을 권장 — 날짜별 파일 분리는 6시간 내 구현 범위에서 제외.
- **출처(`sources`)에는 제목·URL과 함께 `publishedDate`(발표 날짜)·`source`(매체명)를 남긴다.**
- **`generator`** 로 어떤 방식(모드 A/B, provider, 모델)으로 생성됐는지 기록한다.

### 3.6 배포(Publish) 동작 정의

1. 생성 화면에서 브리핑 확인 후 "배포" → 로컬 서버가 `briefings.json`에 항목 추가
2. `git add briefings.json && git commit -m "add briefing YYYY-MM-DD" && git push`
3. push 즉시 GitHub Pages가 최신 데이터를 서빙 (별도 빌드 불필요)

### 3.7 비기능 요구사항

- 실행 환경: 로컬 PC, Python(Flask). `python app.py` 한 줄로 실행.
- Serper 뉴스 검색 기본 파라미터: 한국 증시 대상 시 `gl=kr`, `hl=ko`, 상위 N건(기본 10건) — 필요 시 조정.
- 키는 서버에만 존재하고 브라우저 응답/공개 스냅샷에 포함되지 않는다.

### 3.8 스테이징 보드 & 자동 후보

생성 화면은 배포 전 후보 카드를 다듬는 **스테이징 보드** 형태로 동작한다.

- **자동 후보 생성(진입 시):** 로컬 서버가 Serper News로 **미국 증시 동향 Top3 + 한국 증시 동향 Top3**(총 6장)를 자동 검색해 **모드 A**로 후보 카드를 만든다. (미국=`gl=us`, 한국=`gl=kr,hl=ko` 기본, 필요 시 조정)
- **수동 검색 추가:** 사용자가 기업/키워드로 검색하면 그 결과도 후보 카드로 보드에 **누적**된다.
- **큐레이션:** 각 후보 카드를 **삭제**하거나 **제목/요약/상세를 편집**할 수 있다. (선택 카드를 모드 B LLM 요약으로 강화하는 것도 허용)
- **배치 배포:** 보드에 **남긴 카드들만 한 번에** `briefings.json`에 append + git push 한다.
- **상태 분리:** 스테이징 보드는 배포 전 임시 상태(서버 세션 또는 임시 파일)이며, `briefings.json`에는 **배포 시에만** 반영된다. 따라서 공개 아카이브에는 사용자가 확정한 카드만 나간다.

## 4. Unit 2 — 브리핑 아카이브 (GitHub Pages)

### 4.1 기능 요구사항

| ID | 요구사항 | 설명 |
|---|---|---|
| F2-1 | 카드 인덱스 | `briefings.json`을 읽어 카드 목록으로 렌더링 |
| F2-2 | 카드 표시 정보 | 날짜, 제목, 핵심 요약 표시 |
| F2-3 | 상세 뷰 | 카드 클릭 시 해당 브리핑의 상세 내용 + 출처(제목·URL·발표일·매체) 표시 |
| F2-4 | 정렬 | 최신 브리핑이 상단에 오도록 날짜 역순 정렬 |

### 4.2 구현 방식 (6시간 제약 반영)

- **날짜별 개별 HTML 파일을 생성하지 않는다.** 대신 `index.html` 단 하나에서 JS로 `briefings.json`을 fetch하여 카드 렌더링 + 클릭 시 모달(또는 인라인 확장)로 상세 표시.
- 이유: 페이지를 매번 새로 생성/커밋하는 로직을 없애서, 데이터 파일 하나만 관리하면 되도록 설계 복잡도를 최소화.
- 빌드 도구(Jekyll 등) 사용하지 않음 — 순수 정적 파일(HTML/CSS/JS)만으로 구성.

### 4.3 비기능 요구사항

- **접근성**: 공개 URL(`https://아이디.github.io/저장소명/`)로 누구나 접근 가능
- **반응형**: 5장 UI 요구사항 참조

## 6. 화면 구성 요약

| # | 화면 | 위치 / 공개 여부 | 구성 |
|---|---|---|---|
| 1 | 생성 화면 (스테이징 보드) | `localhost` (비공개) | 진입 시 자동 후보(미국3+한국3) 표시 + 입력폼(기업·키워드) 수동 검색 → 후보 카드 누적 → 삭제/편집 큐레이션 → 남긴 카드 배치 배포. 모드 A/B 선택. |
| 2 | 공개 아카이브 — 카드 인덱스 | GitHub Pages (공개) | `briefings.json` 읽어 카드 목록(날짜·제목·요약), 최신순 |
| 3 | 공개 아카이브 — 브리핑 상세 | GitHub Pages (공개) | 카드 클릭 시 모달/인라인 확장으로 상세 + 출처 표시 |

→ 물리적 페이지는 **생성 1개(로컬) + 아카이브 1개(`index.html`, 인덱스↔상세 뷰 전환)**.

## 7. 사용자 사용 순서

1. `.env`에 키 입력 → `python app.py` 실행
2. 브라우저로 `localhost:PORT` 접속 → **자동으로 미국3+한국3 후보 카드가 뜸**
3. (선택) 기업·키워드로 **수동 검색** → 후보 카드 추가, 모드 A/B 선택 (B면 provider 선택)
4. 불필요한 카드 **삭제**, 필요하면 제목/요약/상세 **편집** (선택 카드 LLM 강화 가능)
5. 남긴 카드를 확인 후 **"배포"** → `briefings.json`에 일괄 append + git push
6. 공개 URL에서 카드 인덱스 자동 갱신 → 카드 클릭 → 상세 확인

## 9. 구현 우선순위

1. **[필수]** Serper 검색 연동 + 결과 정규화 (모드 A 데이터 확보)
2. **[필수]** `briefings.json` append + 아카이브 `index.html` 카드+상세 렌더 (端-to-端 최소 파이프라인)
3. **[필수]** git publish 연결 → 실제 공개까지 관통
4. **[핵심]** 스테이징 보드 UI: 자동 후보(미국3+한국3) + 수동 검색 누적 + 큐레이션(삭제/편집) + 배치 배포
5. **[확장]** 모드 B: LLM 요약 + provider 추상화(Claude)
6. **[확장]** 반응형(모바일 우선) 스타일 마감
7. **[여유]** provider 교체(Gemini 등) 실검증

> 핵심 전략: **모드 A로 "입력→검색→저장→공개"를 먼저 관통**시켜 놓고, LLM(모드 B)을 그 위에 얹는다. 시간이 부족해도 모드 A 아카이브는 항상 동작한다.

## 10. 향후 확장 (Out of Scope, 이번 6시간 범위 제외)

- 자동 스케줄링(매일 아침 자동 실행)
- 날짜별 개별 정적 페이지 생성
- 다중 사용자/인증
- 실시간 주가 데이터 연동
- LLM provider 추가(Gemini 등) 실검증 — 인터페이스는 미리 열어두되 구현은 여유 시
